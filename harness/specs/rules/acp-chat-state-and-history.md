title: ACP Chat State And History Authority
type: ai-coding-rule

Main owns ACP process, SDK, routing lifecycle, and serialization of operations on the shared ACP connection; Renderer owns semantic reduction into an in-memory timeline. Notifications emitted during `session/load` are returned as one generation-scoped raw batch and reduced in one Renderer state commit. Renderer may temporarily buffer matching host events during the IPC result handoff, while ordinary live prompt updates continue through host events. A pending prompt may retain a bounded Main routing context and Renderer timeline snapshot so navigation cannot drop its stream; those contexts must be keyed by session and generation, remain memory-only, and be released when the prompt settles. Permission requests are interactive only for an active prompt. Stale session generations are ignored, and ClawX does not persist a second ACP ledger or reduced Chat history.

Existing-session ACP loads must include explicit session routing metadata so a newer metadata-only OpenClaw replay ledger cannot mask transcript replay. Automatic creation of the default ACP session must wait until a Gateway-ready session discovery pass confirms that the key is absent. Internal OpenClaw image-completion user triggers may participate in compatibility correlation but must not render as user messages.

ACP replay is the primary history authority. Approved supplements are limited to: (1) best-effort recovery of asynchronous image-generation completions with proven `image_generate` context, (2) recovery of explicit line-leading assistant OpenClaw `MEDIA:` attachment directives omitted by ACP, and (3) scheduled-task prompt and completion summaries from Main's typed cron-history host API when ACP replay for that cron session is completely empty. The cron exception must come from Gateway `cron.runs` (with legacy file fallback owned by Main), be generation-scoped and in memory, and must never replace or duplicate non-empty ACP replay. The general attachment exception does not require image-generation context, but it recovers only attachment references. All exceptions remain marked and in memory; do not generalize them to bare paths, surrounding transcript prose, arbitrary ordinary messages, tool cards, plans, permissions, thoughts, file activity, or any parallel persisted history.

The shared historical transcript read is limited to the newest `1000` messages. A successful live prompt reads immediately and retries exactly once after `1500 ms`. A transcript-derived image-generation completion is eligible only when its task id/tool-call id correlates to an ACP image-generation start, when its originating transcript user turn aligns exactly to an active ACP user turn by binary-free prompt projection and duplicate occurrence from the tail, or when it is the single image turn immediately preceding the first exact historical turn match. The boundary exception is disabled if that first ACP turn already has a matched image task and must insert before the replay suffix, never after newer turns. Any other transcript task whose user turn is omitted by ACP replay remains ineligible. General attachment alignment treats that history as a suffix and matches the binary-free OpenClaw prompt-text projection of structured ACP user blocks by duplicate occurrence from the tail; it must not parse or globally remove user-authored resource marker text. Attachment-only empty projections remain eligible, and live alignment also requires the current optimistic user identity. Every asynchronous result must retain the same active session, generation, supplement operation and attempt, and live turn where applicable. Unmatched, ambiguous, superseded, or stale work cannot mutate the timeline.
